Задание 2. Проектирование решения

    Текущие проблемы:
        - Смешение данных: PII, медицинские и финансовые данные хранятся вместе (Excel, общий диск)
        - Отсутствие контроля доступа: Нет RBAC/ABAC, данные доступны всем сотрудникам
        - Небезопасные интеграции: 1С, API лаборатории, ККМ передают данные без шифрования
        - Нет аудита: Нельзя отследить, кто и когда обращался к данным
        - Data Lineage отсутствует: Невозможно понять, откуда взялись данные и как обрабатывались

    Слой аналитики с учётом Privacy by Design:
        - Data Minimization 
          a) В BI-отчётах используются агрегированные данные (без PII)
          b) Вместо «Иван Иванов, 35 лет, диагноз X» → «Пациент M, возрастная группа 30-40, диагноз X»
        - Различия Privacy
          a) Добавление статистического шума в отчёты (чтобы нельзя было идентифицировать человека)
          b) Используется в ML-моделях для обучения на анонимизированных данных
        - Безопасные Data Lakes
          a) Raw Zone: Исходные данные (зашифрованы)
          b) Trusted Zone: Очищенные, но ещё не анонимизированные данные (доступ только для доверенных лиц)
          c) Analytics Zone: Полностью анонимизированные данные для BI и AI
        - Контроль доступа к аналитике
          a) Attribute-Based Access Control (ABAC) (Например: «Только сотрудники с ролью Data Scientist могут видеть агрегированные медицинские данные»)
          b) Динамическое Masking (Если бухгалтер открывает отчёт, ФИО автоматически заменяются на ID)
